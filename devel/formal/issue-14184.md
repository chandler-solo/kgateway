# Issue 14184 formal-methods notes

## Context

Issue [kgateway-dev/kgateway#14184](https://github.com/kgateway-dev/kgateway/issues/14184) reports v2.3.0 upstream connect failures where one pod for a gateway stopped receiving endpoint updates. The report includes go-control-plane warnings like `ADS mode: not responding to request type.googleapis.com/envoy.config.endpoint.v3.ClusterLoadAssignment [...]`, plus kgateway defer logs for per-client endpoints, per-client clusters, and missing referenced clusters.

PR [kgateway-dev/kgateway#13868](https://github.com/kgateway-dev/kgateway/pull/13868) added a per-client readiness gate to avoid publishing snapshots whose routes/listeners reference clusters that are not present yet. That strengthened snapshot safety, but it also made the distinction between "retain last good snapshot" and "eventually publish a newly coherent snapshot" important.

## Formalized failure shape

The issue-focused model is `devel/formal/tla/XdsEdsSubset.tla`.

It models:

- CDS as the set of EDS clusters currently emitted for a client.
- EDS as the set of ClusterLoadAssignments currently emitted for that client.
- Envoy's requested EDS names as the names induced by the current CDS.
- go-control-plane ADS behavior that only responds to a named EDS request when the snapshot's EDS resource names are a subset of Envoy's requested names.

The safe transition removes a cluster from CDS and filters the matching EDS resource at the same time. The buggy transition removes a cluster from CDS while retaining its stale EDS resource.

## Counterexample meaning

`XdsEdsSubsetBug.cfg` intentionally fails `NoOrphanEndpointResources` and `EDSRequestRespondable`:

- `NoOrphanEndpointResources` fails because EDS still contains a ClusterLoadAssignment for a cluster no longer present in CDS.
- `EDSRequestRespondable` fails because Envoy's named EDS request follows current CDS, while the retained EDS snapshot contains a resource outside that request.

This matches the go-control-plane warning shape in the issue: in ADS mode, a named request can be left unanswered when the snapshot contains resources Envoy did not request.

## Concrete validator seam

`pkg/kgateway/translator/xdscheck` now checks the concrete analog:

- Every EDS cluster must have a matching ClusterLoadAssignment.
- Every emitted ClusterLoadAssignment must correspond to an emitted EDS cluster.

The second check is issue-driven. It catches stale or extra EDS resources that are not necessarily missing dependencies, but can still block ADS named EDS responses.

## Root-cause hypothesis

The most plausible root-cause class is not an ACK/NACK or nonce bug. It is a snapshot-set coherence/progress bug:

- A per-client snapshot may be withheld to avoid publishing incomplete CDS/RDS/LDS dependencies.
- The xDS cache intentionally retains the last good snapshot during that defer window.
- If the retained or rebuilt snapshot has EDS resources that no current EDS cluster will cause Envoy to request, go-control-plane can refuse to answer EDS requests.
- If the deferred per-client collection does not later publish a coherent snapshot for that client, the pod can remain stuck with stale endpoint state until restart.

The next production fix should focus on ensuring the EDS resource set is filtered to the EDS clusters in the same per-client CDS snapshot, and on adding a progress-oriented regression test for "backend removed or endpoints changed while per-client snapshot is deferred, then all inputs become coherent and the client receives a new snapshot."
