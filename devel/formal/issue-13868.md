# Issue 13868 formal-methods notes

## Question

PR [kgateway-dev/kgateway#13868](https://github.com/kgateway-dev/kgateway/pull/13868) claims to fix a reconnect-time xDS race where a per-client snapshot could be published while routes and listeners referenced backend clusters that were not yet present in the same CDS snapshot. Envoy could then return transient `NC`/`500` responses until the missing clusters arrived.

The modeling question is whether that readiness gate is correct for its stated property, or whether TLC can still find a partial-snapshot publication under the same assumptions.

## Model

The focused model is `devel/formal/tla/XdsReconnectRace13868.tla`.

It abstracts:

- Two route/listener referenced clusters: `cluster-a` and `cluster-b`.
- A reconnect that starts a fresh per-client computation while the xDS cache retains the previous coherent snapshot.
- Per-client cluster translation becoming ready one cluster at a time.
- Explicitly errored clusters, which PR 13868 allows to satisfy the readiness gate.
- Publishing either with the 13868 gate or with the old non-nil/partial behavior.

The key invariant is:

```text
refs subset cacheClusters union cacheErrored
```

In words: the snapshot currently served from cache must not contain route/listener references to clusters that are neither present in CDS nor explicitly errored.

## Result

The passing config, `XdsReconnectRace13868.cfg`, checks the safe 13868 behavior:

```bash
java -jar /path/to/tla2tools.jar \
  -config XdsReconnectRace13868.cfg \
  XdsReconnectRace13868.tla
```

TLC finds no counterexample. Under the model assumptions, PR 13868 is correct for the narrow property it claims: it prevents a new per-client snapshot from overwriting the retained coherent snapshot until every referenced cluster is present or explicitly errored.

The intentionally failing config, `XdsReconnectRace13868Bug.cfg`, checks the old partial behavior:

```bash
java -jar /path/to/tla2tools.jar \
  -config XdsReconnectRace13868Bug.cfg \
  XdsReconnectRace13868.tla
```

TLC produces the expected counterexample:

- Initial state: the retained cache snapshot has both clusters, so route references are coherent.
- `ClusterReady("cluster-a")`: only one referenced cluster has finished per-client translation.
- `BuggyPublish`: the server overwrites the cache with a snapshot containing only `cluster-a`.
- The invariant fails because `cluster-b` is still referenced but is neither in CDS nor explicitly errored.

That counterexample supports the PR's root cause and explains why the readiness gate is needed.

## What This Does Not Prove

This model does not prove the full xDS implementation, Envoy behavior, or Kubernetes watch ordering.

It also does not prove issue 14184 is fixed by PR 13868. The 14184-focused model, `XdsEdsSubset.tla`, checks a different property: EDS resources must be filtered to the EDS clusters in the same CDS snapshot. That later model shows why 13868 can be correct for missing referenced clusters while still being insufficient for stale or extra EDS resources.

## Conclusion

The model supports this statement:

> PR 13868 is correct for preventing reconnect-time publication of snapshots whose route/listener cluster references are missing from CDS unless the missing cluster is explicitly errored.

The model does not support this stronger statement:

> PR 13868 proves per-client xDS publication is fully safe or rules out all ADS progress failures.
