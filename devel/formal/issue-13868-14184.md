# Combined 13868 and 14184 per-client publication model

## Question

Issues around PR [kgateway-dev/kgateway#13868](https://github.com/kgateway-dev/kgateway/pull/13868) and issue [kgateway-dev/kgateway#14184](https://github.com/kgateway-dev/kgateway/issues/14184) share a per-client publication theme:

- During reconnect or controller startup, kgateway should retain the last coherent per-client xDS snapshot instead of overwriting the cache with a partial snapshot.
- Once inputs become coherent, kgateway should publish a new snapshot whose CDS and EDS resource sets are mutually compatible.
- Envoy should not activate a new snapshot unless the EDS response is compatible with the CDS names Envoy has learned.

The combined model asks whether those properties hold together for the two minimal failure traces.

## Model

The focused model is `devel/formal/tla/XdsPerClientPublication.tla`.

It models:

- Desired route/listener cluster references separately from the last snapshot stored in the xDS cache.
- A retained last-good cache snapshot.
- A reconnect phase where per-client clusters rebuild as empty, partial, then coherent.
- The PR 13868 allowance that an explicitly errored referenced cluster satisfies the publication gate.
- A stale-EDS phase where `cluster-b` is removed from CDS while its `ClusterLoadAssignment` remains in endpoint inputs.
- A minimal Envoy view with known CDS/RDS/LDS state, active state, warming state, and named EDS response behavior.

The model is phase-based on purpose. It does not try to enumerate every possible Gateway edit. It exhaustively checks the two target traces with the relevant publication and Envoy activation interleavings.

## Safe Invariants

The passing config, `XdsPerClientPublication.cfg`, checks:

- `CacheSnapshotCoherent`: the cache snapshot has no missing referenced clusters, no missing endpoint resources for referenced EDS clusters, and no orphan EDS resources.
- `EnvoyActiveSnapshotCoherent`: Envoy's active snapshot remains coherent even while a newer snapshot is warming or deferred.
- `AlignedEDSRequestRespondable`: once Envoy's known CDS matches the cache CDS, the cache EDS resource set can answer Envoy's named EDS request.
- `CoherentInputsCanPublish`: when inputs are coherent and differ from the cache, a safe publish is enabled.

## Counterexamples

`XdsPerClientPublicationMissingClusterBug.cfg` models the pre-13868 publish rule: publish once any per-client cluster result exists.

TLC finds the expected trace:

- Start from a coherent snapshot with `cluster-a` and `cluster-b`.
- Reconnect clears per-client cluster and endpoint inputs.
- Only `cluster-a` becomes ready.
- The buggy publisher overwrites the cache with CDS containing only `cluster-a`.
- The cache still has route/listener references to `cluster-b`, so `CacheSnapshotCoherent` fails.

`XdsPerClientPublicationStaleEdsBug.cfg` models publishing with the cluster readiness gate but without filtering stale EDS resources.

TLC finds the expected trace:

- Start from a coherent snapshot with `cluster-a` and `cluster-b`.
- Desired config removes `cluster-b`, and CDS now contains only `cluster-a`.
- Endpoint inputs still contain stale EDS resources for both clusters.
- The buggy publisher writes EDS `{cluster-a, cluster-b}` with CDS `{cluster-a}`.
- `CacheSnapshotCoherent` fails because `cluster-b` is an orphan EDS resource. If Envoy has learned the new CDS, the named EDS request is also not respondable.

## Result

The safe combined model supports this statement:

> Retaining the last coherent per-client snapshot, gating publication on referenced cluster and endpoint readiness, and filtering EDS to the same CDS snapshot rules out the two minimal failure traces behind 13868 and 14184.

The model does not prove all xDS startup and warming behavior. It intentionally keeps Envoy warming abstract and does not model Kubernetes watch ordering, all Envoy resource types, or full go-control-plane cache internals.

## How to Run

Run the passing combined model:

```bash
cd devel/formal/tla
java -jar /path/to/tla2tools.jar \
  -config XdsPerClientPublication.cfg \
  XdsPerClientPublication.tla
```

Run the two intentional counterexamples:

```bash
cd devel/formal/tla
java -jar /path/to/tla2tools.jar \
  -config XdsPerClientPublicationMissingClusterBug.cfg \
  XdsPerClientPublication.tla

java -jar /path/to/tla2tools.jar \
  -config XdsPerClientPublicationStaleEdsBug.cfg \
  XdsPerClientPublication.tla
```
